# ADR-006: Pico 2W Custom Flight Controller over Navigator/Pixhawk + ArduSub

## Status
Accepted

## Context
Initial hardware spec used a Navigator Flight Controller running ArduSub — a proven, supported platform. A more affordable alternative was identified: build a custom flight controller on a Raspberry Pi Pico 2W running MicroPython.

## Decision
Custom flight controller on Pico 2W for v1 bench build. Navigator/Pixhawk + ArduSub retained as a potential future upgrade path for v2. The redundancy and sophistication of a Pixhawk/Navigator is not required for minirov v1.

minirov's initial deployment profile is:
- Calm, sheltered water (lake shore or harbour)
- Shallow depth (<10m)
- Short duration missions (<30 minutes)
- Operator present and tethered at all times
- No safety-critical payload

In this context, a complementary filter provides sufficient attitude estimation. Basic PID loops provide sufficient stabilisation. The vehicle does not need an EKF, redundant sensors, or hardware failsafes to be safely deployable — the tether and operator presence are the primary safety mechanisms. Additional hardware+software safety checks will be performed prior to deployment. 

- Primary goal of the project is engineering learning — writing a flight
  controller from scratch teaches IMU integration, sensor
  fusion, PID control, and PWM generation at the lowest level
- Pico 2W is low cost — acceptable loss if hardware is damaged
  during development
- Architecture is conceptually identical to ArduSub:
  sense → estimate → control → actuate. The difference is
  engineering depth, not fundamental approach.
- SITL work validates the MAVLink layer independently —
  the flight controller is a separable, swappable concern
- Direct continuity: understanding the custom FC makes
  ArduSub's internals more interpretable when working
  with professional systems

## Consequences
No ArduSub failsafes, no EKF, no MAVLink from the FC in v1. These are accepted limitations for a bench/shallow watermlearning platform. 

The Pico does not speak MAVLink natively. This has  consequences for the communication architecture — see ADR-008.
