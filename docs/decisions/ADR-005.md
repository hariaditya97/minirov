# ADR-005: ROS 2 Jazzy as Middleware Layer

## Status
Accepted — supersedes ADR-002

## Context
ADR-002 established plain Python as the v1 middleware on the basis of simplicity. The architecture subsequently expanded:ma separate Ubuntu machine (Lenovo) was introduced as a dedicated middleware node between the Mac (LLM/operator) and the Pico FC. This created a natural ROS 2 deployment
boundary and removed the original justification for avoiding it.

## Decision
Lenovo runs Ubuntu 24.04 with ROS 2 Jazzy as the middleware layer. minirov-brain on the Mac communicates with ROS 2 nodes on the Lenovo over the local network. ROS 2 is the de facto standard middleware for marine and underwater robotics. Using it from the earliest possible stage, even before wet hardware exists is a deliberate choice to build the right foundations rather than refactor later.


- Every Python module in minirov-brain maps directly to a ROS 2 node. The plain Python v1 architecture was written with this migration in mind. 
- ROS 2's DDS transport layer provides automatic node discovery, typed message passing, and quality-of-service guarantees across the Mac/Lenovo network boundary — capabilities that would require significant custom engineering to replicate in plain Python.
-  ROS 2 nodes are independently testable and replaceable. The LLM layer, the serial bridge
to the Pico, and the telemetry aggregator are fully decoupled. This matters as the system grows in complexity.

## Consequences
Increased setup complexity compared to plain Python — accepted as the cost of building on the right foundation. ROS 2 Jazzy on Ubuntu 24.04 is well supported and the tooling has matured significantly. The complexity is front-loaded into setup, not ongoing development.
