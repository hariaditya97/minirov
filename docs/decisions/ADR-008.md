# ADR-008: Pico 2W Communication Protocol — JSON Serial Bridge

## Status
Accepted 

## Context
The Pico 2W flight controller does not speak MAVLink natively. The minirov stack, specifically minirov-brain on the Mac and ROS 2 on the Lenovo was originally designed around MAVLink,
reflecting the initial Navigator/ArduSub hardware spec - See ADR-006.

With the Pico as the v1 flight controller, a communication protocol between the Pico and the ROS 2 middleware layer must be defined. Three options were considered:

1. Implement MAVLink on the Pico in MicroPython
2. JSON over USB serial, with a ROS 2 bridge node on the Lenovo
3. MAVLink router on the Lenovo translating JSON to MAVLink

## Decision
JSON over USB serial between Pico and Lenovo. A dedicated ROS 2 node on the Lenovo acts as a serial bridge — reading JSON telemetry from the Pico and publishing ROS 2 topics, and translating incoming ROS 2 command messages into JSON serial writes to the Pico.

**Implementing MAVLink on the Pico is premature for v1.**
The MicroPython MAVLink implementation requires generating
and trimming dialect files to fit within the Pico's 264KB
RAM, handling byte stream framing, sequence numbers, and
checksums at 50Hz. This is significant additional complexity
before the basic flight controller has been validated on
real hardware in water. The right time to implement MAVLink
on the Pico is after v1 is proven — or not at all if the
upgrade to Navigator + ArduSub happens first.

**JSON serial is already implemented and tested.**
The Pico flight controller already outputs JSON telemetry
and accepts JSON commands. This protocol is working in
Wokwi simulation. Reusing it avoids introducing new
failure modes before bench testing.

**The ROS 2 bridge node is a clean abstraction.**
minirov-brain on the Mac communicates with ROS 2 topics
and has no knowledge of the underlying transport. When
the stack upgrades to Navigator + ArduSub in v2, the
bridge node is replaced with a MAVLink node — nothing
else in the stack changes.

**MAVLink is preserved where it belongs.**
The SITL pipeline on the Mac retains full MAVLink
functionality. minirov-brain's mavlink.py remains valid
for SITL testing and for the potential Navigator upgrade path in v2.
The JSON bridge is an isolated adapter, not a replacement
for MAVLink across the whole stack.

## Consequences
minirov-brain on the Mac does not talk MAVLink to the Pico in v1. It talks ROS 2 to the Lenovo, which talks JSON serial to the Pico.

The mavlink.py files in minirov-brain remain valid for:
- ArduSub SITL testing
- Navigator + ArduSub v2 upgrade

The upgrade path from Pico to Navigator is clean:
replace the ROS 2 JSON bridge node with a ROS 2 MAVLink
node. The rest of the stack is unchanged.

This architecture should be reviewed if the Pico is
retained beyond v1 — at that point implementing
MAVLink on the Pico becomes the right investment.
